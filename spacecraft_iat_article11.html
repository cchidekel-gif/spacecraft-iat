Ask an engineer what is aboard a spacecraft at any point during integration and test, and the answer should be immediate, precise, and verifiable: this part number, this serial number, at this location, installed per this drawing revision, verified by this inspection record, changed from the previous configuration by this approved Engineering Change Notice. If the answer is "I think it's the latest version" or "we changed that out last week, I believe," the program has a configuration management problem — and configuration management problems have a way of becoming anomaly investigation problems, and sometimes mission-loss problems, at the worst possible moment.

Configuration management is among the least glamorous disciplines in spacecraft I&T. It requires a general, broad technical understanding but not the technical depth by a subsystem engineer. Its tools are largely administrative. And yet it is the discipline that makes every other discipline's work meaningful: the test result is only interpretable if you know exactly what configuration of hardware was under test. The anomaly investigation can only identify root cause if you know exactly what changed between the last good test and the first bad one. The Launch Readiness Review can only close its verification matrix if you know exactly what you intend to launch.

"Configuration management is the answer to a question that never stops being asked: what, exactly, are we working with right now.? The question is asked when you start a test, when you find an anomaly, when you ship to the launch site, and when something goes wrong on orbit. If you cannot answer it quickly, precisely, and with documented evidence, everything else stops. Anomaly investigations stall. Review boards cannot assess risk. Launch readiness arguments collapse. Not knowing your configuration is not a paperwork shortfall — it is an engineering failure that consumes time and money at exactly the moments when the program can least afford either."

1. What Configuration Management Actually Is

Configuration management (CM) is the technical and administrative discipline that establishes and maintains the consistency between a product's documented description and its physical realisation. For a spacecraft, it encompasses four interrelated activities, each of which is essential and none of which can substitute for the others.

  • Configuration identification — the definition and documentation of the spacecraft's configuration at each stage of its development: what the design requires, what was actually built, and how the two relate. This includes the part numbering and serialisation system, the drawing tree, the bill of materials, and the specification hierarchy.
  • Configuration control — the process by which changes to the established configuration are proposed, evaluated, approved or rejected, implemented, and verified. No change to the spacecraft's design, hardware, specifications, requirements, other documents or software is made outside this process. The Configuration Control Board (CCB) is the governing authority for configuration control.
  • Configuration status accounting (CSA) — the continuous recording and reporting of the current status of configuration items: which version of each component is installed, which changes have been approved and implemented, which are pending, and which are planned. CSA is the living record of the spacecraft's current configuration at any given moment.
  • Configuration audits — the formal verifications, conducted at defined program milestones, that the physical hardware and software are consistent with the documented configuration and that the documentation is complete, accurate, and current. The Functional Configuration Audit (FCA) and Physical Configuration Audit (PCA) are the two primary audit types for spacecraft programs.
⚙ CM Is Not Bureaucracy

Configuration management is frequently perceived — particularly by early-career engineers and by engineers trained primarily in technical disciplines — as administrative overhead: paperwork that slows things down without adding engineering value. This perception is understandable and completely wrong.

The purpose of CM is to ensure that the program always knows what it has. The value of that knowledge is invisible when everything goes right. It becomes acutely visible when something goes wrong. When an anomaly is found during a functional test, the first question the investigation asks is: what changed since the last good test? That question is only answerable if the CM record is current, complete, and accurate. When a spacecraft anomaly is observed on orbit, the same question is asked of the pre-launch configuration record. Every anomaly investigation in this series' history — every one — began with a configuration question. The programs with rigorous CM answered it in hours. The programs without it answered it in days, if they answered it at all.

2. The Configuration Baseline Sequence

A configuration baseline is a formally approved description of the spacecraft's configuration at a specific point in the program — a snapshot that serves as the reference against which subsequent changes are measured and controlled. Baselines are established at formal program reviews and represent the program's collective agreement that the configuration as documented at that point is technically correct and approved for the next phase of development. Once established, a baseline can only be changed through the formal change control process.

Phase A/B
SRR
Functional Baseline
What the spacecraft must do — mission requirements and system-level performance specifications
The highest-level description of what the spacecraft must do — the mission requirements and system-level performance specifications. Established at System Requirements Review (SRR). Defines the "what" without yet specifying the "how." All subsequent design work flows from this baseline. Changes to the functional baseline require the highest level of CCB authority and have the broadest downstream impact.
Phase B/C
PDR
Allocated Baseline
Decomposition of system requirements into subsystem and component specifications
The decomposition of system-level requirements into subsystem and component specifications — the allocation of performance requirements to specific hardware elements. Established at Preliminary Design Review (PDR). Defines what each subsystem must do and how its performance is verified. Changes to the allocated baseline require CCB review and may require re-analysis of requirement margins across multiple subsystems.
Phase C/D
CDR
Product Baseline
Complete design documentation — the "build-to" package
The complete design documentation for the spacecraft as it is to be built — detailed drawings, specifications, software design documents, interface control documents. Established at Critical Design Review (CDR). This is the "build-to" baseline: the complete set of instructions for fabricating and assembling the spacecraft. After CDR, every drawing change, every specification update, every ICD revision is a formal change requiring CCB approval and may require re-verification of affected requirements.
Phase D
PCA / PSR
As-Built Baseline
What was actually built — every component, serial number, and change recorded
The documented record of the spacecraft as it was actually built — every component installed, every serial number recorded, every drawing revision verified as the basis for the physical hardware, every post-CDR change incorporated and documented. Built progressively throughout the I&T phase and formally audited at the Physical Configuration Audit before launch. This is what was launched. It is the primary reference for anomaly investigation throughout the mission lifetime.
Phase E
Mission Life
Operational Baseline
As-built baseline plus all on-orbit changes — the living configuration record
The as-built baseline as modified by on-orbit changes — software patches, parameter updates, operational procedure changes — accumulated over the mission lifetime. Every in-flight change is documented against the as-built baseline to maintain an accurate current-configuration record. The operational baseline is the reference for on-orbit anomaly investigation and for mission extension planning.

3. Configuration Identification — Naming, Numbering, and Serialisation

Before any component can be controlled, it must be uniquely identified. Configuration identification is the system of part numbers, drawing numbers, serial numbers, revision levels, and lot codes that uniquely identifies every configuration item in the spacecraft from the top-level assembly down to individual fasteners and connectors.

The Part Numbering System

A part number identifies a design — a specific item with a specific drawing and a specific specification. All hardware built to the same part number must be physically interchangeable. A part number change is required whenever the design changes in any way that affects form, fit, or function. The distinction between a change that requires a part number change and a change that does not is one of the most consequential judgements in CM: an incorrectly classified change — one that changes the hardware's performance but is not reflected in a part number change — makes it impossible to determine from the CM record alone whether a given installation has the old or new version of the part.

A serial number identifies a specific physical instance of a part — a specific piece of hardware built to a specific part number. Two components with the same part number are design-equivalent but physically distinct; their serial numbers distinguish them. The serial number is the hook on which the component's entire life history is hung: its acceptance test data, its handling history, its installation records, its anomaly history. Losing track of a serial number means losing track of that history.

Revision Control

Every drawing and specification in the product baseline is issued at a specific revision level. When the document is changed, its revision level advances. The CM record must associate every installed component not just with its part number but with the drawing revision to which it was fabricated and inspected. A component fabricated to drawing revision B may differ from a component fabricated to drawing revision C even if the part number is the same — drawing revisions below a part-number-change threshold represent clarifications, tolerance refinements, or manufacturing process changes that do not affect interchangeability but do affect the exact physical characteristics of the part.

◆ The "It's Just a Minor Change" Problem

The most dangerous words in configuration management are "it's just a minor change." A change that appears minor from a design perspective — a wire re-route, a connector substitution, a software parameter update — may have consequences that are not apparent from the change description alone. The CM process exists precisely to ensure that every change, however minor it appears, is reviewed by people with the breadth of knowledge to assess its full impact.

One program I worked on had a "minor" change to a harness connector — the connector was no longer manufactured by the original vendor and was replaced with an electrically equivalent part from a different manufacturer. The replacement was electrically correct, it would mate properly with the componet where it was to be connected. What the change assessment had not evaluated was the new manufacturer had a different outer shell diameter: it was about 1 milimeter larger than the original. In the spacecraft's closeout configuration, with thermal blankets and adjacent brackets, there was precisely enough clearance for the original connector but not for the replacement. The interference was not discovered until pre-thermal testing closeout — after the connector was installed on flight hardware. The resulting rework cost six weeks and several thousand dollars. The change assessment that approved the connector substitution had not included a physical clearance check. "It's just a connector change" had been assumed to be benign.

4. The Configuration Control Board

The Configuration Control Board (CCB) is the formal governance body responsible for evaluating and approving — or rejecting — all proposed changes to the configuration baseline. It is not an advisory committee. Its decisions are program-binding: no change to a baseline document or to flight hardware proceeds without CCB approval, and the CCB record is part of the configuration audit trail. The CCB's membership, authority levels, and decision process are defined in the program's CM plan.

Engineering Change Request — CCB Process Flow
1
Engineering Change Request (ECR) Initiated
Any program member may initiate an ECR to propose a change to a baselined document or to flight hardware. The ECR must identify: the specific item being changed (part number, document number, revision); the reason for the change (anomaly correction, improvement, supplier issue, test finding); the proposed change description; and an initial assessment of the change's impact on interfaces, mass, power, and verification status.
2
Impact Assessment
The CCB assigns the ECR for technical impact assessment. The assessment must cover: all documents affected by the change; all hardware items affected; all requirements whose verification may be affected; mass, power, link budget, and thermal margin impacts; schedule impact; cost impact; and the effect on any currently open DRs. The assessment is performed by the affected engineering disciplines, not by the person who originated the ECR.
3
CCB Review and Decision
The CCB reviews the ECR and its impact assessment at a formal board meeting. The board may approve, reject, or request additional information. Approval requires that the change is technically sound, that all impacts have been adequately assessed, that any required re-verification is identified and planned, and that the change is within the board's authority level. Changes above the board's authority level are escalated to the next-higher CCB tier which is often the Program Manager or Chief Engineer.
4
Engineering Change Notice (ECN) Released
An approved ECR is released as an Engineering Change Notice — the formal authorization to implement the change. The ECN is the work order for the change: it specifies exactly what is to be done, which documents are to be revised, which hardware items are to be modified, and what verification is required to close the change. No implementation of the change begins before the ECN is released. Implementing a change before CCB approval is a serious CM violation regardless of how technically straightforward the change appears.
5
Implementation and Verification
The change is implemented per the ECN. Required re-verification activities are executed and their results documented. All affected documents are updated to their new revision levels. The hardware traveller records the ECN number, the work performed, and the verification results. The CSA database is updated to reflect the new configuration.
6
ECN Closure and CSA Update
The ECN is formally closed when all required implementation and verification actions are complete and documented. The CCB confirms closure. The as-built configuration record is updated. The change is now part of the established baseline. Any future changes to the same item start a new ECR — they do not informally reopen the closed ECN.
◆ The ECN closeout Problem

Since configuration cortrol is a shared technical / administrative function it is not uncommon for the engineering change to move forward in parallel with the paperwork. This can become a problem especially later in the program where schedule pressure is mounting.

One program I worked on had found a major design flaw associated with cryogen boil off and vent paths. The error was caught in a milestone technical review and a plan to remedy was presented and approved by the review board. The relevant paperwork was generated however, due to schedule pressure, implemeting the change was delayed and eventually forgotten. The instrument was launched with the un-remedied flaw and failed as expected, less than a month into the mission. Why was this not caught in a subsequent pre-launch review ? The ECN was closed based on the "PLAN" to inplement the change, not on implementation and verification of the change. So in the LRR, the issue was listed as closed because the paperwork said it was closed. For effective configuration management and control engineering and administative personnel must work hand-in-hand.

CCB Authority Levels

Not every change requires the full program-level CCB. Most programs use a tiered authority structure: subsystem-level CCBs have authority to approve changes within their subsystem that have no impact outside it; the system-level CCB has authority for changes with cross-subsystem or interface impacts; changes that affect the mission baseline, a customer-controlled document, or a safety-critical requirement require escalation to the highest authority level — typically the chief engineer and program manager, with customer concurrence for NASA and ESA missions.

The tiered structure is efficient but carries a risk: a change that appears to be within a subsystem's authority may have impacts outside that subsystem that the subsystem team does not see. The impact assessment process — which must explicitly evaluate cross-system impacts — is the safeguard against this risk. A subsystem-level CCB that approves a change without verifying that it has no interface impacts is not performing adequate CM — it is merely performing expedient CM.

5. Configuration Status Accounting — The Living Record

Configuration Status Accounting (CSA) is the ongoing record of the spacecraft's current configuration: a continuously updated inventory of every configuration item, documenting its current part number, serial number, drawing revision, installation location, and the complete history of changes that brought it to its current state. CSA is not a one-time document — it is a database that must be updated with every change, every installation, every removal, and every verification activity.

CSA Element What It Records Why It Is Essential
Bill of Materials (BOM) Every component in the spacecraft assembly — part number, quantity, reference designator, installation location — for every assembly level from top to bottom. The BOM links the design drawing tree to the physical hardware inventory. The BOM is the map of the spacecraft. Without it, there is no systematic way to confirm that the physical hardware matches the design intent or to identify every item affected by a specific design change.
Serialised Part Record For every serialised component — every component with a unique serial number — the record of its part number, serial number, acceptance test status, installation location, handling history, and anomaly history. Enables tracking of every specific physical part from manufacture through integration to launch. Essential for anomaly investigations that require identifying which specific hardware instance was involved, and for fleet-wide anomaly assessment when a defect is found in one serial number that may affect others.
Drawing and Document Revision Status The current revision level of every drawing, specification, ICD, and procedure in the product baseline. Updated whenever a document is revised through the CCB process. Ensures that anyone using a document to perform work, verify hardware, or plan a change is using the current approved version. A technician working from a superseded drawing revision is building or inspecting to the wrong specification.
Open and Closed ECN Log Every Engineering Change Notice — approved, in work, or closed — with its status, implementation record, and verification closure evidence. The ECN log is the complete change history of the spacecraft from the product baseline forward. Without it, there is no way to determine, for any given item, what the original baseline configuration was and what changes have been made since.
Verification Matrix Status The current status of every requirement verification — open, in work, or closed — and the identity of the specific test, analysis, inspection, or demonstration that provides the closure evidence. The verification matrix status is the live record of the program's progress toward launch readiness. Every change that reopens a previously closed requirement must be reflected immediately in the matrix status.
DR/ECN Linkage The explicit association between Discrepancy Reports and the ECNs that address them — ensuring that every DR that requires a hardware or software change has an associated, approved ECN, and that every ECN that was initiated to close a DR has the DR formally closed in the DR system upon ECN implementation. Prevents DRs from being closed by reference to ECNs that were never fully implemented, and prevents ECNs from being implemented without their associated DRs being formally closed. The linkage is the structural connection between the anomaly management system and the configuration management system.

6. The Traveller — Documenting Every Hand That Touches the Hardware

The traveller — also called a router, a work order packet, or a build record depending on the program — is the document that accompanies a specific piece of hardware through every step of its assembly, test, and inspection history. It is the operational heart of the as-built configuration record: the place where the action of building and testing the spacecraft is recorded in real time, by the people performing the work, with signatures and timestamps that create a legally and technically auditable chain of custody.

Anatomy of a Hardware Traveller — Key Elements
Hardware Identity
Part number, serial number, drawing revision, and lot code of the item being built or tested. Verified at the start of every operation. If the hardware identity on the traveller does not match the hardware in hand, work stops immediately pending resolution.
Work Instructions
Reference to the specific procedure, drawing, and revision that governs each step. Not paraphrased — referenced by document number and revision so that the exact instructions are traceable. Steps are marked complete only when the specified procedure has been executed as written, not as interpreted.
Step Sign-Off
Each step is signed and dated by the person who performed it. For critical or irreversible steps, a second signature from an independent inspector or QA representative is required. No step may be retroactively signed — signatures are contemporaneous with the work. Blank signature blocks at a final review indicate incomplete work.
Torque Records
For fasteners with specified torque requirements — structural fasteners, electrical connector retaining screws — the actual torque applied is recorded, not simply "torque applied per drawing." The calibration certificate number of the torque tool used is recorded. Untorqued connectors and fasteners have caused anomalies in flight; the torque record is the evidence that the specified torque was applied.
Materials Used
Lot numbers and expiry dates of all materials — adhesives, potting compounds, lubricants, conformal coatings — used during assembly. Expired or out-of-certification materials may not be used on flight hardware. Recording the lot number enables traceability if a material later shows a quality issue.
Anomaly Notation
Any observation that deviates from the expected — a connector that is harder to seat than normal, a torque value that is at the low end of the specified range, a visual indication that is different from the acceptance standard photograph — is documented in the traveller and entered as a DR. The traveller is not the place to decide that an anomaly is acceptable; it is the place to record that it was observed.
Test Data Reference
References to test data files and test reports generated during testing of the item. The traveller does not contain the test data itself — it contains pointers to where it is stored, with document numbers sufficient to retrieve the data. The test report reference, combined with the serial number, creates the link between the physical item and its performance record.
Cleanliness and ESD
Cleanroom class, contamination survey results, and ESD precaution compliance recorded for every operation conducted in a controlled environment. The contamination record is the evidence that the cleanroom protocol was followed; it is also the first reference when a contamination anomaly is discovered later in the program.

The traveller's authority derives entirely from the accuracy and completeness with which it is maintained. A traveller with blank fields, retroactive entries, or pencilled corrections is a traveller that cannot be trusted — and a traveller that cannot be trusted is a gap in the as-built record that must be resolved, not accepted. Programs that treat traveller maintenance as a formality to be completed after the fact rather than a discipline to be performed concurrent with the work produce as-built records that are reconstructions rather than records.

7. Configuration Audits — FCA and PCA

◆ A note about audits

The FCA and PCA terminology originates from MIL-HDBK-61B and is standard in NASA and DoD programs. ESA/ECSS programs accomplish the same functions through the Qualification Review (QR) and Acceptance Review (AR). Commercial programs may use terms such as Verification Close-Out, As-Built Configuration Verification, Acceptance Data Package Review, or First Article Inspection (FAI) for what are fundamentally the same activities: confirming that the hardware meets its requirements (FCA function) and confirming that the documentation matches the hardware (PCA function). The names vary; the engineering necessity does not.

Configuration audits are the formal verification events at which the program confirms that its documentation accurately describes what it has built. There are two primary audits for spacecraft programs, each with a distinct scope and purpose.

Functional Configuration Audit (FCA)

The Functional Configuration Audit (FCA) verifies that the spacecraft's performance has been demonstrated to meet every requirement in the functional and allocated baselines — that the verification matrix is complete and that every requirement has been closed by a passing test result, analysis, inspection, or demonstration. The FCA is essentially an audit of the verification record: it confirms that the test program was executed as planned, that the results are documented and traceable, and that any requirements verified by analysis are supported by approved analysis documentation with sufficient detail to reproduce the analysis.

The FCA is typically conducted at or near the Pre-Ship Review — after all primary test activities are complete but while there is still time to address any deficiencies identified during the audit. An FCA finding that a requirement has no verified closure evidence is a serious program issue: it means the program cannot formally demonstrate that the spacecraft meets that requirement. The finding must either be resolved by executing the required verification activity or by formally re-baselining the requirement as acceptable at its current verified state — neither of which is a quick or simple resolution.

Physical Configuration Audit (PCA)

The Physical Configuration Audit (PCA) verifies that the physical spacecraft matches its documented design — that every component installed in the spacecraft corresponds to an approved part number and revision, that the serial numbers match the CSA database records, that all approved ECNs have been implemented and closed, and that no unapproved changes have been made to the flight configuration. The PCA is the formal closing of the as-built record.

A PCA is conducted by physically comparing the spacecraft to its documentation — the BOM, the drawing tree, the ECN log — item by item. For a large spacecraft, this is a substantial activity: the audit may cover hundreds of line-replaceable units, thousands of individually tracked components, and dozens of ECNs from the I&T phase. The PCA team typically includes the CM lead, the quality assurance lead, the systems engineer, and representatives from each major subsystem. The audit produces a PCA report that documents any discrepancies found and their disposition — resolved before launch or formally accepted with documented rationale.

⚠ The PCA Finding That Changes the Launch Date

The PCA is scheduled before the Pre-Ship Review, which is itself weeks before launch. This timing is not coincidental — it is designed to give the program time to resolve PCA findings while the hardware is still at the "home" facilty. This has the necessary equipment and GSE to make corrections, before shipping to the launch site where these resources are not available or difficult to arrange. A PCA finding that reveals an unapproved change, an incorrect serial number, or an unimplemented ECN requires investigation and resolution. In the worst case, it requires physical disassembly of the spacecraft to access and correct the affected component.

PCA findings that affect the launch date are not program failures of the PCA process — they are exactly what the PCA process is designed to produce. The failure mode the PCA prevents is launching a spacecraft whose physical configuration is unknown or differs from the approved design. A PCA finding that is resolved before launch — even at the cost of schedule — is far less costly than the equivalent uncertainty discovered during an on-orbit anomaly investigation.

8. Software Configuration Management

Flight software is a configuration item with its own CM requirements that parallel, and in some respects exceed, the requirements for hardware CM. The flight software build — the specific compiled binary that executes aboard the spacecraft — is the software equivalent of the as-built hardware record, and it must be managed with corresponding rigour.

The principles of software CM are the same as hardware CM: every change requires CCB approval; the current version is always known and accessible; a complete history of all versions and changes is maintained; and the version loaded on flight hardware is verified against the released build by checksum comparison. Article 08 covers the software build and version control process in detail. What belongs here is the integration between software CM and hardware CM — the requirement that the software configuration record and the hardware configuration record together constitute a complete and consistent description of the flight system.

  • Software version is a required element of the as-built record. The as-built configuration of the spacecraft is not complete unless it records the specific software build loaded into every processor aboard the spacecraft — the flight computer, the attitude control processor, each instrument's microcontroller, the solid-state recorder, the RF subsystem's firmware. Each of these is a software configuration item with its own version identifier that must appear in the CSA database alongside the hardware serial numbers.
  • Software changes during I&T require the same CCB process as hardware changes. A software update — even a "parameter update" that only changes a coefficient in a table — is a change to the flight configuration. It requires an ECR, an impact assessment, CCB approval, an ECN, implementation verification, and CSA update. The instinct to treat software changes as inherently less consequential than hardware changes is one of the most reliable predictors of software-related anomalies in the late I&T phase.
  • The as-built software configuration must be verified immediately before launch. The last act of the pre-launch functional test is to read back the software version identifiers and checksums from every processor and compare them to the approved flight build. This verification is not a formality — software can be inadvertently overwritten, corrupted, or replaced with an incorrect version at any point in the launch site processing sequence. The pre-launch verification is the confirmation that what is loaded is what was approved.
⚠ Don't ignore EGSE and ground test software.

The same configuration discipline applies to EGSE and ground test software — and the consequences of neglecting it are deceptively severe, because EGSE software errors masquerade as spacecraft problems. When a functional test produces an anomalous result, the first question the anomaly investigation asks is: was the test setup correct? If the EGSE software version is not controlled, baselined, and recorded in the test report, the team cannot answer that question. They cannot distinguish a spacecraft defect from a test artifact — and they will spend days reproducing the anomaly, re-running the test, and interrogating the flight hardware before discovering that the EGSE command script was modified between test runs by someone who did not realize the change would affect the telemetry interpretation. Every EGSE software element — command scripts, telemetry processing databases, automated test sequences, limit-check definitions, and simulator models — must be version-controlled, change-managed through the same CCB process as the flight software, and recorded by version number in every test procedure and test report. The version of the EGSE software is part of the test configuration, and the test configuration is part of the test result."

9. When CM Breaks Down — The Failure Modes

🔀
The Unapproved Change
An engineer or technician makes a change to the hardware or software without going through the CCB process — a wire re-route "to make integration easier," a connector substitution "because the original is out of stock," a software parameter update "to fix an obvious bug." The change may be technically correct. But it is not in the CM record. It will not appear in the PCA. It will not be visible when the anomaly investigation asks "what changed?" The unapproved change is the most common CM failure mode and the one most likely to directly cause investigation delays and incorrect root cause conclusions.
📦
Serial Number Confusion
Two components of the same part number are interchanged during a maintenance operation, and the serialised part records are not updated. The CSA database now records the wrong serial number at each location. When one of the components is later found to have a performance anomaly in its test history, the investigation cannot determine from the CSA record whether the anomalous component is the one in location A or location B. Resolution requires physical inspection and potentially disassembly. In programs with many serialised components, serial number confusion during rework is a persistent source of CSA errors that requires disciplined incoming and outgoing inspections at every maintenance operation.
📝
The Retroactive Traveller
Work is performed without contemporaneous traveller sign-off — technicians complete multiple steps and sign them all at the end of the work session, or a week after the work was done. The traveller record no longer reflects what was done; it reflects what the technician remembers or believes was done. This is indistinguishable in the record from correct practice unless the timestamps reveal the anomaly. A traveller that was signed in bulk at the end of a day provides no evidence that each step was actually performed when and as specified. Some programs have found, during PCA, that traveller entries were backdated — a discovery that invalidates the entire as-built record for the affected hardware.
🏃
The "Obvious" Change
A change that appears so obviously correct, so trivially minor, that the engineer implementing it judges it unnecessary to go through the CCB process. This failure mode is correlated with schedule pressure: when the program is behind, the CCB process is perceived as a bottleneck, and small changes begin to be implemented without CCB review. Each individual bypass may be technically defensible. Collectively, they erode the integrity of the configuration record and remove the oversight mechanism that catches the occasional change that is not as obviously benign as it appears. The connector substitution story in §3 is a canonical example.

10. The As-Built Record — What It Contains and Why It Lasts

The as-built record — the complete, audited documentation of the spacecraft's configuration at the time of launch — is one of the most valuable and most permanent products of the I&T program. It does not stop being useful at launch. It is the primary reference for anomaly investigation throughout the entire mission lifetime, for mission extension planning, and for the design of future missions that will inherit technology or lessons from this one.

A complete as-built record contains:

  • The final BOM at every assembly level, with part numbers, serial numbers, drawing revisions, and installation locations — a complete map of what is aboard the spacecraft.
  • The complete ECN log from the product baseline forward — every change that was made during development and I&T, with its technical rationale, impact assessment, and implementation verification record.
  • The closed verification matrix — the formal record that every requirement was verified, with references to the specific test reports, analysis documents, or inspection records that provide the closure evidence.
  • All test reports and test data from the complete I&T program — unit test reports, environmental test reports, functional test reports, calibration data — indexed and retrievable by configuration item and test date.
  • The closed DR log — every anomaly observed during I&T, its root cause, corrective action, and formal closure, including the waivers and deviations that were accepted. This is the anomaly history of the spacecraft; it is the first reference when an on-orbit behaviour resembles something that was seen on the ground.
  • Software configuration records — the specific build identifiers and checksums of every software item loaded aboard the spacecraft at launch, with their V&V records.
  • The PCA and FCA reports — the formal audit records that confirm the as-built record was reviewed, found consistent, and accepted by the program's technical authority.
✓ The As-Built Record in Action — On-Orbit Anomaly Investigation

A spacecraft experiences an unexpected thermal anomaly during its second eclipse season — a component temperature that is 8°C higher than the thermal model predicted. The operations team's first call is to the I&T archive: what did this component read during the thermal-vacuum test? What was the thermal model prediction at that time? Were there any DRs against this component during T-V testing? Was there an ECN that changed anything in the thermal path near this component?

The answers to all of these questions are in the as-built record — but only if the record was maintained with the discipline the article describes. A program with a complete, searchable, accurate as-built record answers these questions in hours. A program without one answers them in weeks, if at all, and may never achieve the diagnostic certainty needed to confidently resolve the anomaly. The thermal anomaly is a hypothetical. The diagnostic process is not — it is the actual process used in every on-orbit anomaly investigation for every mission currently operating.

11. Configuration Management as Engineering Discipline

Configuration management is not a quality assurance function imposed on the engineering team from outside. It is an engineering discipline — a set of practices that the engineering team applies to its own work because those practices are necessary to know what has been built, to trust the test results, and to investigate anomalies effectively. The CM plan is not the quality manager's document; it is the engineering team's plan for maintaining the integrity of the information on which every technical decision depends.

The early-career engineer who treats CM as paperwork to be completed after the real engineering work is done will, eventually, be the engineer trying to reconstruct the configuration history of a spacecraft component whose anomaly has no apparent explanation — and discovering that the explanation is buried in a change that was made without CCB review, implemented without a traveller entry, and accepted without a verification signature. That engineer will learn, at that moment, what configuration management is for. The purpose of this article is to make that lesson available without the discovery cost.

Four principles, concisely:

  • If it isn't written down, it didn't happen. The traveller entry, the ECN, the DR, the test report — these are not descriptions of what happened. They are what happened, in the only form that is accessible to anyone who was not physically present and to you yourself six months from now when your memory of the detail has faded. The discipline of documentation is the discipline of creating a reliable institutional memory.
  • Every change goes through the CCB, without exception. The CCB is not a bureaucratic obstacle between the engineering team and the right answer. It is the mechanism that ensures people with the relevant knowledge — systems engineers, interface owners, safety representatives — have the opportunity to identify impacts that the person making the change may not have considered. Every bypass of the CCB process, however well-intentioned, removes that opportunity.
  • The CM record is only as good as the most recent audit. A configuration record that has not been audited against the physical hardware is a record of intent, not of fact. Regular audits — not only the formal PCA but interim configuration audits during the I&T campaign — are the mechanism by which the CM record is kept honest. An audit that finds discrepancies is performing its function. An audit that finds none should prompt the question: was this audit thorough enough?
  • The as-built record is a mission-lifetime investment, not a launch-day deliverable. The effort to create and maintain a complete, accurate, retrievable as-built record is an investment that pays returns throughout the spacecraft's operational life. It is not complete at launch — it is a living record that continues to be extended with on-orbit changes. The program that maintains this record faithfully provides its operations team with an asset that is, in the long run, as valuable as any instrument aboard the spacecraft.

Key Terms Used in This Article

TermDefinition
Allocated BaselineThe approved decomposition of system-level requirements into subsystem and component specifications. Established at PDR. Changes require CCB review and may require re-analysis across multiple subsystems.
As-Built RecordThe complete, audited documentation of the spacecraft's physical and software configuration at the time of launch — including BOM, ECN log, verification matrix, test records, DR log, and software configuration records.
Configuration BaselineA formally approved description of the spacecraft's configuration at a specific program milestone, serving as the reference against which subsequent changes are measured and controlled.
Configuration Control Board (CCB)The formal governance body responsible for evaluating and approving or rejecting all proposed changes to a configuration baseline. Its decisions are program-binding; no change proceeds without CCB approval.
Configuration Status Accounting (CSA)The continuously updated record of the spacecraft's current configuration — part numbers, serial numbers, drawing revisions, ECN status, and verification matrix status — for every configuration item.
Engineering Change Notice (ECN)The formal authorisation to implement an approved change, released by the CCB after ECR approval. Specifies exactly what is to be done, which documents are to be revised, and what verification is required.
Engineering Change Request (ECR)The formal proposal to change a baselined document or flight hardware configuration item. Initiates the CCB review process. Must include identification of the item being changed, reason for change, change description, and initial impact assessment.
Functional Configuration Audit (FCA)The formal audit that verifies the spacecraft's performance has been demonstrated to meet every requirement in the functional and allocated baselines — that the verification matrix is complete and all requirements are closed.
Functional BaselineThe highest-level description of what the spacecraft must do — the mission requirements and system-level performance specifications. Established at SRR.
Part NumberThe identifier for a design — a specific item with a specific drawing and specification. All hardware built to the same part number must be physically interchangeable. Changes that affect form, fit, or function require a part number change.
Physical Configuration Audit (PCA)The formal audit that verifies the physical spacecraft matches its documented design — every component corresponds to an approved part number and revision, all serial numbers match the CSA database, and all approved ECNs have been implemented.
Product BaselineThe complete design documentation for the spacecraft as it is to be built. Established at CDR. Every drawing change after CDR requires CCB approval and may require re-verification.
Serial NumberThe identifier for a specific physical instance of a part — a specific piece of hardware built to a specific part number. The hook on which the component's entire life history — test data, handling, installation, anomalies — is hung.
TravellerThe document that accompanies a specific piece of hardware through every step of its assembly, test, and inspection history. Signed contemporaneously by the person performing each step. The operational core of the as-built configuration record.
